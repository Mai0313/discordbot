# CLAUDE.md

Project-specific guidance for Claude Code in this repository. Dense and AI-agent-facing on purpose: an index of pointers, cross-cutting invariants, and non-obvious pitfalls, not a re-explanation of what the code already documents. When a fact already lives in a code comment or docstring, this file keeps only the pointer plus the load-bearing "why". `AGENTS.md` and `GEMINI.md` are symlinks to this file, so edit `CLAUDE.md` only.

## Commands

All tooling runs through `uv`.

```bash
uv run discordbot                # run the bot
uv run pytest                    # tests, coverage gate: 80% (--cov-fail-under)
uv run pre-commit run -a         # canonical pre-push check (= make fmt)
make gen-docs                    # regenerate docs/ from sources
```

When debugging, check `./data/logs` for the full runtime logs.

## Runtime Shape

- `src/discordbot/cli.py` defines `DiscordBot(commands.Bot)`; intents are `Intents.all()` minus `members` and `presences`.
- Cogs load synchronously in `__init__` before the gateway connects. Every cog module exposes a sync `def setup(bot)` and adds the cog with `override=True`; an `async def setup` breaks the first command sync (nextcord schedules it without awaiting). See `_load_cogs_sync`.
- Helper packages live in sibling `_<cog>/` dirs so the `cogs/*.py` loader glob does not pick them up as cogs.
- Bot presence is hardcoded in `status_task`; there is no DB-backed rotation.
- Add common command errors in `DiscordBot.on_command_error`, not in each cog.

## Cog Rules

- One `commands.Cog` subclass plus a sync `setup` per module. Cogs do not import peer cogs; reach across via the bot instance, shared `typings/` / `utils/`, or cog-private `_<cog>/` packages.
- User-facing slash commands need localized name and description for English, `Locale.zh_TW`, and `Locale.ja`.
- Any user-visible command or behavior change must update `cogs/help.py` in the same change and keep `tests/test_help.py` green (it asserts every non-help slash command name appears in every localized help body).
- For aligned-width embeds use `utils.discord_embeds.embed_spacer_payload(..., target=)` so an edit reuses the uploaded spacer; re-uploading it every edit trips Discord error 400009 on rapidly edited messages (see its docstring).
- Content past embed limits is paginated or rendered to a PNG via `attachment://...`; never truncate silently.
- Never hardcode a Discord attachment-size ceiling: read the destination's real limit through the shared `utils/media_delivery.py::upload_limit_for(guild=...)` (it returns `guild.filesize_limit` so a boosted server's 50/100MB is honored, falling back to 10MB only when there is no guild — DMs expose no per-DM limit). `filesize_limit` is nextcord's boost-tier table lookup keyed on `premium_tier`, not a live field, so it over-reports 25MB for unboosted tier 0/1 (Discord lowered the real base to 10MB in 2024); the guild path keeps trusting it (self-corrects when nextcord refreshes its table). The multipart-envelope margin (`MEDIA_ENVELOPE_MARGIN`, 1 MiB) is held back by `MediaDeliveryPlanner.plan` for a combined edit / an attach riding with embeds (`_gen_reply/streaming.py` and `parse_threads.py` pass it; the single-file sites pass 0). Call sites: `parse_threads.py`, `video.py`, `_gen_reply/streaming.py`, `gen_reply.py`, `_research/delivery.py`.
- **Oversize → URL fallback / unified delivery** (`utils/media_delivery.py`): the attach-vs-host-vs-drop *decision* is unified in `MediaDeliveryPlanner.plan`, which returns a `MediaPlan` (`native` attachments / `hosted_urls` / `dropped_items`). Each feature stays pure: it produces `MediaItem`s (a `source` of in-memory `bytes` OR an on-disk `Path`, plus a `filename` carrying the allowlisted suffix) and hands a list to `plan`; only the actual Discord send + the per-site degradation policy stay local (the sends are structurally disjoint, so unifying them buys nothing). A clip/file larger than `upload_limit_for` is written to an external static host (`MediaHostingService.publish_bytes` / `publish_path`, unguessable name, allowlisted suffix only — mirrors the nginx allowlist, rejects e.g. `.aiff`) and its public URL posted unwrapped (Discord inline-plays it under ~100MB, else a browser link) instead of failing/dropping. Hosted files are **content-addressed** (`<sha256(content)[:32]>.<ext>`): identical bytes dedup to one file (a re-host refreshes its mtime + returns the same URL), written atomically (sibling `.tmp-*` then `os.replace`) so a crash never leaves a poison cache entry. The serve dir **self-cleans** (`MediaHostingService.enforce_cap` size-cap LRU + `cleanup_expired` age-cap + stale-temp sweep, all under a module `threading.Lock`; a recency grace window stops a concurrent publisher reaping another's just-posted URL): each publish enforces `MEDIA_HOSTING_MAX_BYTES` (8 GiB, under R2's 10 GB free tier) eagerly, and the `media_cleanup` cog runs the age (`MEDIA_HOSTING_RETENTION_HOURS`, 168) + size + temp sweep on a `@tasks.loop` + once at startup. The **reaper only ever deletes the bot's own files** — `_HOSTED_NAME_RE` (`^[0-9a-f]{32}\.<ext>$`, ext built from `_ALLOWED_SUFFIXES`, `re.fullmatch`, `is_file(follow_symlinks=False)`, non-recursive) — never a foreign `access.log` / human-named clip in the serve dir. The bot never creates the serve dir (a container-local dir nginx can't see would only 404): a missing dir logs and falls back to host-free delivery, so the docker-compose mount is opt-in, not required for `up`. The size cap + reaper are local-FS-only; a future R2 backend replaces them with a server-side lifecycle rule (content-addressing carries over as the object key). Best-effort: `MediaHostingConfig` (`MEDIA_HOSTING_*`) self-disables when unconfigured (`available`) or when the serve dir is absent, so an oversize item lands in `dropped_items` and **every site degrades to its exact pre-hosting behavior** — `MEDIA_HOSTING_ENABLED=false` (or no serve dir) is byte-for-byte the host-free path. `video.py` dropped its 480p-downgrade retry; the streamer hosts oversized voice/music/inline-image (peeling the largest to a URL when the combined body overflows); `parse_threads.py` hosts only the oversize videos and attaches the rest (per-item, mixed). Wired into IMAGE/VIDEO routes (`gen_reply._deliver_generated_media`, oversize posts a standalone URL message so the persona stream never clobbers it; an unhostable oversize still raises to keep the route's hard-fail path), `video.py`, `parse_threads.py`, the QA streamer, and `_research/delivery.py` (an oversize `research.md`/`.png` is linked instead of silently dropped).

## AI Pipeline

### Backend and invariants

- Runtime LLM backend is LiteLLM Proxy behind `OPENAI_BASE_URL`; all runtime conversations use `AsyncOpenAI` and the OpenAI Responses API. Do not switch back to Chat Completions; do not import provider-native SDKs (`google-genai`, `anthropic`) into request paths. The proxy indirection is the point: one OpenAI SDK for the whole project while the backend stays mostly Gemini and any model is hot-swappable via the model string + `extra_body`.
- **Non-proxied paths** are the only runtime calls direct to Google (each builds `genai.Client(api_key=gemini_api_key)` inline as a `cached_property`, no `create_*_client` factory — the old `utils.llm` ones are gone, see `research.py` for the pattern). They forgo proxy-side cost/usage tracking: Gemini Files API attachment upload (`_gen_reply/attachment/gemini_file_api.py`), the YouTube Interactions answer turn (`interactions.py`), omni video generation / editing via the Interactions API (`_handle_video_reply`), the inline `<music>` Lyria render via the Interactions API (`MusicGenerator`, `generation.py`), and deep-research agents (`cogs/_research/`).
- Runtime model strings live in `RuntimeModelCatalog` (`typings/models.py`); each property's docstring carries its callers and tier rationale (`route_model` flash-lite, `tool_model` flash for nickname-to-id matching, `slow_model` keeps a deliberate same-model peak/off-peak split, `prompt_model` flash+high+grounding for the IMAGE/VIDEO prompt director, decoupled from `slow_model` so its tier tunes alone).
- Author runtime prompts in English (`prompts.py` / `server_prompts.py`, tool descriptions, routing/memory/voice prompts, `TTS_STYLE_DIRECTIVE`); reply *content* is localized at runtime by the in-prompt "follow the user's language" rule, not by writing the prompt in another language. Deliberate exceptions: the few load-bearing Chinese behavior notes in `COMMON_PROMPT` (e.g. the 轉帳 caution); the bot's own name 破貓.
- LLM SDK objects are named `responses`; loop items are named `response`.
- LiteLLM's docs lag its code — answer proxy-behavior questions (request translation, `extra_body`, tool/grounding quirks) from source at `~/repo/litellm/`, not docs.
- For Discord embeds prefer `media.discordapp.net` `proxy_url` over origin URLs (CDN links expire).

### Reply pipeline (`gen_reply` orchestration)

- **Dual render from one history walk**: `_fetch_history` does the single channel-history read whose raw side also feeds the memory allowlist. A text-only twin (`_render_history(text_only=True)` for history + `_get_reference_and_current(text_only=True)` for reference/current, `[attachment: kind]` markers, no upload) feeds routing + memory selection; the uploaded-`file_id` render (`_render_history(text_only=False)` + `_get_reference_and_current()`) is awaited only by the answer, so the route never blocks on the Files API. `_prepare_reply_context` starts memory selection on the text-only renders and awaits the uploads concurrently, so selection overlaps the upload window instead of running after it.
- **Route is two parallel calls under one `route_done` gate**: `_route_classify` returns `RouteClassification` (only critical-path call, gates dispatch); `_grade_effort` returns `EffortGrade` concurrently. QA/SUMMARY override the slow model's effort with the grade (footer shows `model (effort)`); IMAGE/VIDEO cancel the effort task. Memory selection + effort grading share the `route_done`-gated `_await_gated` helper (unbounded while the route runs, then only a grace: `MEMORY_SELECT_GRACE_SECONDS` / `EFFORT_GRACE_SECONDS`). On timeout/error effort falls back to `high`; memory selection falls back to `_participant_memory_fallback` (author + reply-reference authors, allowlist-gated) rather than dropping memory; a selection that deliberately picked nobody injects nothing.
- **Speculative `ReplyContext`** (`_prepare_reply_context`): history + server memory + memory selection built concurrently with the route. IMAGE/VIDEO consume it (not cancel) — awaited only after the media is on screen so its build overlaps generation; the reply injects `memory_block` but never `server_memory_block`. SUMMARY rebuilds at `history_limit=100`, `memory_enabled=False`, reusing shared parts.
- **Streaming UX** (`streaming.py`): progress is reactions on the user's message (`ReactionStatusChain`, ordered, fire-and-forget, custom app emojis); no separate status messages. `ResponseStreamer` previews the reasoning summary as `-#` subtext, replaced by the first content delta. Edits run on a snapshot editor task that shuts down via a stop-event (not cancellation) so an in-flight write always lands before the final edit (else it orphans the reply). `_reply_or_send` degrades `message.reply` to `channel.send` on a deleted source (Discord 50035 / `NotFound`).
- **Best-effort convention** (voice, inline image, image/video persona replies): any failure leaves the primary deliverable plus a ⏱️/⚠️ react hint on the source, never raising into the pipeline's outer error path. An oversize clip/image is hosted as a URL first (the decision lives in `utils/media_delivery.py`; see the oversize → URL fallback in Cog Rules); only an unhostable one falls back to the drop + ⚠️ hint.
- **Kill-switches** (`LLMConfig`, runtime; off still strips any marker but produces nothing): `youtube_video_enabled`, `deep_research_enabled` / `deep_research_max_enabled`, `inline_voice_enabled`, `inline_image_enabled`, `inline_music_enabled`, `inline_video_enabled` (the last two gated through `music_available` / `video_available`, which also require `gemini_api_key`, like `deep_research_available`). `refine_prompt_enabled` is the odd one out (gates the IMAGE/VIDEO `PromptGenerator.refine` step, not a marker; off sends the raw request straight to the model).
- **Logfire spans**: `gen_reply pipeline` / `route` / `effort` / `context build` / `memory selection` / `answer` (carries a `backend` attr) / `image reply` / `video reply` / `threads context`, plus first-reasoning-delta + first-content-delta logs in `streaming.py`.

### Routes

- **QA** (default): answer streams with the built-in `googleSearch` / `urlContext` tools; carries the inline voice/image/music markers and the deep-research marker.
- **SUMMARY**: rebuilds at `history_limit=100`, no memory, no selection. A URL-bearing SUMMARY is rerouted to QA by `_route_classify`.
- **IMAGE** (`_handle_image_reply`): the raw request is first expanded by `PromptGenerator.refine` (`prompt_model` flash+high + grounding, `IMAGE_PROMPT`, kill-switch `refine_prompt_enabled`, best-effort so a director failure falls back to the raw prompt) then renders via `ImageGenerator.render` (an empty prompt falls back to a Chinese default; one retry on an empty result before raising). Edit needs raw pixels, so it loads bytes via `get_image_source_bytes` (not Files-API handles) and passes them as input images — to both the refine (as grounding) and the render. Image sent first; then a best-effort persona reply (`IMAGE_REPLY_PROMPT`, `image_reply_model` flash+low, no tools) via the shared `_stream_media_persona_reply` (history + selected user memory, never server memory), so a reply failure never disturbs the delivered image.
- **VIDEO** (`_handle_video_reply`): native Gemini omni via `VideoGenerator.render` (`interactions.create` DIRECT to Google, the Interactions API is Gemini-only; the single downstream video render in `generation.py`, render-only since video is always the primary deliverable). 16:9 for text / reference generation (an edit omits the aspect ratio — omni 400s it — and keeps the source clip's ratio), `delivery="uri"` (the clip returns as a Files URI, no base64 bloat, downloaded via `_download_output_video` — a bounded download-retry, NOT an ACTIVE poll: `files.get` can report `PROCESSING` right after the interaction is `completed` even though `files.download` already serves the finalized bytes, so retrying the download directly is both correct and lower-latency than waiting for ACTIVE; the file id is the uri's last path segment before the `:download` suffix); duration is omni's default. Three input modes by precedence: a source video on the message / replied-to message is uploaded to the Files API and edited in place (`get_video_sources` raw bytes, `task="edit"`, the raw request as the literal edit instruction, and the prompt director is SKIPPED — it only grounds on image parts, so a video-only edit would run it blind and add latency; omni takes a single input here so accompanying reference images are dropped); else the request is expanded by `PromptGenerator.refine` (`VIDEO_PROMPT`, flash+high+grounding) and up to `MAX_VIDEO_REFERENCE_IMAGES` message + replied-to images ride as inline subject references (`task="reference_to_video"`); else plain text (`task="text_to_video"`). omni signals a soft refusal / incomplete / budget_exceeded by returning `status != "completed"` with no `output_video` (there is no `error` / `rai_*` field on the interaction), so `render` folds `status` + `output_text` into the raised `RuntimeError`. Bounded by `VIDEO_RENDER_TIMEOUT_SECONDS`. Clip first; then best-effort `_reply_about_video` → shared `_stream_media_persona_reply` (`video_reply_model` flash watches it via a Files-API `input_file` through the proxy).
- **YouTube `watch_video`** (`interactions.py`, Gemini-only, kill-switchable): the Responses bridge can't make Gemini watch a YouTube video (LiteLLM `convert_url_to_base64` fetches the URL as HTML), but the native Interactions API forwards the uri untranslated for server-side fetch (verified against the LiteLLM interactions transform). On a YouTube URL (`utils/youtube.py::YOUTUBE_URL_RE`) + `RouteClassification.watch_video` (QA-only gate), `_handle_message_reply` swaps that one answer turn to `create_interactions_answer_stream` (drops `extra_body` + the LiteLLM end-user header); `to_interactions_input` / `adapt_interactions_stream` translate input + stream so the streamer/footer/memory/marker path is reused unchanged (usage emitted once on `interaction.completed` so the `+=` over the selection seed never double-counts). Only backend swap; no second pass.
- **Deep research** (`cogs/research.py` + `cogs/_research/`, **direct to Google**, kill-switch `deep_research_enabled`): minutes-long cited reports on Gemini managed agents via the Interactions API. Direct because the proxy drops `agent_config` and 404s the Files API upload, and Antigravity `background=True` works despite docs — all spike-verified. Trigger: QA-only `<deep-research>brief</deep-research>` marker (`DEEP_RESEARCH_INSTRUCTION`) + `/deep_research`. Tiers in `RuntimeModelCatalog`: `antigravity_model` (default), then `deep_research_model` / `deep_research_max_model` (`ResultEscalationView` -> native `collaborative_planning`; Max gated by `deep_research_max_enabled`). Persisted in `reply.db` table `research`; `on_ready` resumes (`store=True` keeps it alive server-side). One per owner; `gen_reply.on_message` skips research-thread messages so QA never double-handles a plan-refine turn. Report language follows the request; status words stay English.
    - **Streamed reasoning** (`_research/streaming.py` + `agent.py::_StreamDriver`/`_drive`): every interaction (research, plan/refine, resume) runs `stream=True`, so the agent's `thought_summary` deltas paint live onto the thread's opening status message via `ResearchProgressStreamer` (a self-contained windowed-`-#` cadence editor mirroring `_gen_reply/streaming.py`, NOT reusing `ResponseStreamer`). `_StreamDriver` captures the id on `interaction.created` (persisted via an `on_created` callback before the long wait), tracks `event_id` as the resume token, and re-attaches a dropped stream with `interactions.get(stream=True, last_event_id=...)`; resume re-attaches an already-running run with `get(id, stream=True)`. A full streaming failure degrades to `_poll_until_terminal` (`_latest_thought` stays for that path). The final result is ALWAYS read from a terminal non-stream `get(id)` -> `_to_result` because `interaction.completed` carries an empty payload on purpose, so streamed deltas are the live view only and delivery (`delivery.py::deliver_report`) is unchanged.
- **Threads context** (`THREADS_URL_RE`, `utils/threads.py`): on a Threads URL the current message carries, `gen_reply` self-parses it metadata-only (`parse_metadata`, no download) via `build_threads_context_messages` on a `route_done`-gated task (`THREADS_GRACE_SECONDS`), independent of the `parse_threads` cog. Injects text + image/video as URL parts (raw signed Threads CDN URLs the proxy fetches server-side, verified on Gemini) into `ReplyContext.threads_block`. Text is injected unconditionally (private/deleted -> a "could not read it" notice, never a hallucinated body), so the model never says "I cannot open this link" — the failure mode of the reverted #294. Provider-aware (non-Gemini gets text); first URL only; IMAGE/VIDEO cancel the task.

### Inline markers (`markers.py`, `voice.py`, `generation.py`)

- The answer model self-selects, inside its own output, which spans to speak (`<voice>...</voice>`, kept visible) and which images / music / video to attach (`<image>...</image>`, `<music>...</music>`, `<video>...</video>`, removed from visible text so the prompt never leaks). Markers may repeat where it makes sense: every `<voice>` span is concatenated into one clip and every `<image>` block becomes its own attachment, while `<music>` and `<video>` are single-per-reply (first non-empty kept). The decision stays inside the answer model so text, clip, and media stay coherent (a post-hoc classifier wouldn't). `extract_inline_markers` pulls them all at finalize; `scrub_markers_for_preview` hides tags from the preview (image/music/video hidden as they stream, voice kept). Both mask markdown code regions (fenced and inline-backtick spans) to NUL sentinels before matching and restore them after (`_mask_code_regions` / `_restore_code_regions`), so a literal `<video>` shown as example code is never pulled: `<video>` is the one marker whose name collides with a real HTML element (`<img>` / `<audio>` are the HTML tags, not `<image>` / `<music>`), and the prompts already forbid a real marker from sitting in code, so anything inside code is example content by construction. All run after the text is on screen (zero added latency) and ride a single `Message.edit(files=[...])` carrying the voice clip plus every other attachment — `edit` replaces the attachment list, so two edits would drop the earlier files. None adds a function tool on the answer turn, so none risks the grounding silent-drop.
- **Voice** rides QA+SUMMARY (`allow_voice`, `COMMON_PROMPT`): the wrapped segments are concatenated in order into one `VoiceSynthesizer` input, no spoken-length cap by design (the prompt tells the model multiple spans stitch into one clip, so it tags only the worthwhile lines instead of the whole reply). TTS quirks in `voice.py` docstring: the proxy ignores `instructions`, 500s on `response_format` (style rides in the input prefix, output WAV), single fixed `TTS_VOICE`. **Image** is QA-only (`allow_image`, `REPLY_PROMPT` + `INLINE_IMAGE_INSTRUCTION`): each `<image>` description is rendered best-effort by `ImageGenerator.generate` (twin of the IMAGE route's `render`, one implementation); up to `MAX_INLINE_IMAGES` (9, so the voice clip + images fit Discord's 10-attachment cap) render concurrently in `_build_image_candidates` and ride the one edit, extras hard-dropped. When the user uploaded image(s), `_build_image_candidates` loads their bytes (`_load_marker_source_bytes`, via the streamer's `input_builder`, mirrors the IMAGE route's current + replied-to gather) and passes them so the `<image>` **edits** them instead of generating fresh. Unlike the IMAGE route, the marker description is NOT run through `PromptGenerator` (the answer model already authored it with grounding), so `refine_prompt_enabled` does not affect it. Deliberately separate from the IMAGE route (image-first UX); routing decides which runs first so they never double-fire. IMAGE/VIDEO routes never speak or inline-image. **Music** is QA-only too (`allow_music`, `REPLY_PROMPT` + `MUSIC_INSTRUCTION`, gated on `music_available`): one `<music>` block per reply (first non-empty, single clip by design) renders best-effort via `MusicGenerator.generate` (direct-to-Google Lyria on the Interactions API, non-streaming) in `_build_music_candidate`; the style defaults to Japanese anime / J-pop *with Japanese lyrics* (Lyria picks the lyric language from the prompt, so the language is defaulted, not just the genre; steered in the prompt, reinforced by `MUSIC_STYLE_DIRECTIVE`) but the model departs for a genre/language the user asks for or an instrumental. The clip rides the same single edit; its extension follows the returned audio mime (`music_filename`). Unlike the voice clip, the bot's own music attachment is NOT skipped from later history input (`input.py` skips only `reply.wav`): the `<music>` description is stripped from the visible reply, so the clip is the only trace of the song and is deliberately re-ingested so a later turn can reference it. **Video** is QA-only too (`allow_video`, `REPLY_PROMPT` + `VIDEO_INSTRUCTION`, gated on `video_available`): one `<video>` block per reply (first non-empty, single clip by design) renders best-effort via `VideoGenerator.generate` (the best-effort twin of the VIDEO route's raising `render`, one implementation, direct-to-Google omni on the Interactions API) in `_build_video_candidate`. When the user attached image(s) they ride as subject references (`task="reference_to_video"`, reusing `_load_marker_source_bytes`); otherwise plain text-to-video (source-video in-place editing stays the VIDEO route's job). `VIDEO_INSTRUCTION` caps it at ONE clip per reply as a plain capability limit, NOT a cost warning (telling the model a clip is expensive makes it over-refuse). Almost every clip exceeds Discord's upload limit, so `MediaDeliveryPlanner` hosts it as a URL; the shared `source_bytes_task` loads uploaded pixels once even when `<image>` and `<video>` both fire, since `load_image_bytes` re-fetches per call.

### Attachments (`_gen_reply/input.py`)

- Ingestion is gated by `get_supported_modalities` for the slow model (`MessageInputBuilder._supported_sources`); unsupported attachments are dropped before any LLM call. Images downscale to 3072px longest edge (`utils.images.shrink_image_bytes`) first.
- **Forwarded messages**: a Discord forward leaves `content`/`embeds`/`attachments` empty and puts the original payload in `message.snapshots` (`List[MessageSnapshot]`, stickers as `sticker_items`). `get_cleaned_content` folds each snapshot's text in tagged `[forwarded message]` (appended, so a forwarder's own comment is kept), and `collect_attachment_sources` collects each snapshot's media via the shared `_sources_from_parts`, so a forward is never blank. Detection is just non-empty `message.snapshots` (no `reference.type` check); a normal message has none, so behavior is unchanged.
- **Provider-aware** on `_answer_model_is_gemini` (`"gemini" in slow_model.name`): a non-Gemini answer model can't resolve a Gemini Files URI (the proxy mistranslates it), so its attachments are inlined per type by `_inline_file_part` (image base64, PDF base64, UTF-8 text, else dropped; needs no `GEMINI_API_KEY`). Gemini is the default (also ingests video, hits the inline cap sooner): it uploads every supported attachment to the Files API and references the full `uri` as an `input_file` `file_id`, never base64 (avoids the per-part `inline_data` cap; the bare `files/<id>` name fails the proxy's mime lookup).
- Upload/adopt mechanics are in `input.py` docstrings: `_upload_file`'s 15s ACTIVE-poll + the deprecated-`uploaded`-status / "not in an ACTIVE state" 400, `PendingUpload` / `_resolve_file_upload` re-poll-and-adopt with a lazy `load_data` (no re-download), and the per-message LRU cache (keyed on message id, edit time, `collect_attachment_sources` ids). Media fetch + upload is bounded by `MEDIA_CONCURRENCY` (`_media_semaphore`, shared on the singleton builder so concurrent pipelines don't starve each other); URL fetches use a 10s timeout; a failed-fetch source is skipped for `DEAD_SOURCE_TTL` so an expired-CDN attachment in scrollback isn't re-fetched and re-warned every reply.

### Responses API Gotchas

- Strict role/content part pairing: `user` / `system` / `developer` use `input_*` parts; `assistant` uses `output_text` or `refusal`. So a message with attachments falls back to `role=user` (see `_assemble_input_message`). Prefer string-content shorthand when there are no attachments.
- The top-level `instructions` parameter has `developer` authority and outranks everything in `input`, but does not persist across turns. Put behavior rules there (or in `COMMON_PROMPT`), never in an input-side wrapper.
- Separator messages use `role=system`, not `developer`, for Gemini and Claude compatibility through LiteLLM.
- When a custom function tool rides with the built-in `googleSearch` / `urlContext` in one Gemini request, LiteLLM silently drops the grounding (search annotations go to zero, no 400). So when mixing tool kinds, verify the grounding still fires, not just that the call succeeds. The Gemini 3 opt-out (`include_server_side_tool_invocations: true` flat in `extra_body`) is Preview-only, so memory retrieval stays two-phase on purpose: the selection request carries only the function tool, the answer carries only the built-in tools.
- The code-execution built-in tool is intentionally omitted from `ModelSettings.tools`: providers validate every uploaded file part against a narrow MIME allowlist and 400 the whole request on video / audio / GIF-as-video, so it can't coexist with the Files-API attachment path. The search/url tools have no such limit.
- Gemini prepends leading newlines to the first reasoning and content delta; `_on_reasoning_delta` / `_on_content_delta` strip them. Gemini thought summaries only flow through the Responses API.
- `client.images.edit(image=...)` needs raw `bytes`, not image-reference dicts.
- Per-token pricing comes from `utils.model_pricing` and its cached LiteLLM JSON; do not hardcode rates.

## Economy

- DB files under `data/database/`: `messages.db` (log_msg), `economy.db` (per-user 虛擬歡樂豆, jackpot pools, casino ledger), `games.db` (Blackjack history, fishing state, cleanup targets), `stock.db`. `_games` / `_fishing` / `message_cleanup` each keep their own engine on the shared `games.db`; intentional.
- `_economy/database.py` owns the module-level `_engine` (its docstring has the pool-leak + test-monkeypatch reason); do not move it to `cached_property`.
- `UserAccount` has no `guild_id` (identity, VIP, check-ins, flags are cross-server). Money columns are `StoredInteger` decimal text parsed to Python `int` to dodge SQLite's 64-bit ceiling: balance in `user_wallet`, loans in `loan_proposal` / `loan_contract`, daily casino counters in `casino_account`.
- Invariant: every applied positive delta bumps `total_earned`, every negative bumps `total_spent`, so `total_earned - total_spent == balance`. There is no transaction table.
- Money inputs are string `SlashOption`s parsed via `utils.amount_parsing.parse_decimal_amount`, wrapped per domain (`_parse_positive_amount` / `_parse_collect_amount`, `parse_wager_amount`, `parse_bait_quantity`); malformed text returns `build_invalid_amount_embed` ephemerally before any mutation. Reuse for any numeric input that can exceed Discord's int cap.
- `UserAccount.avatar_url` is a last-seen cache via `utils.avatars.guild_avatar_url`; do not backfill existing URLs.
- `credit_with_repayment` is the single income facade (message reward, casino payout); passive income and `/give` recipients do not auto-repay debt.
- Loans: personal loans debit the lender on acceptance; central-bank loans mint on approval and burn on repayment / collection (no lender credit). Proposals expire after `LOAN_PROPOSAL_TIMEOUT_SECONDS` (180s). Acceptance prepays `MIN_INTEREST_DAYS` of interest, so do not zero `interest_due` at contract creation.
- `is_central_banker` (set offline by direct DB write) is separate from Discord admin; keep `ECONOMY_ALLOW_CENTRAL_BANK_SELF_APPROVAL` unset or `false` in production.
- Admin tweaks go through `adjust_balance(..., allow_negative=...)`, never casino settlement helpers (it skips loans and daily casino counters).
- `/leaderboard` sorts `StoredInteger` text DB-side with `LIMIT` before rows reach Python (`top_n`); `hide_from_leaderboard` accounts are omitted; write paths must call `invalidate_economy_leaderboard_cache()`. `/loss_leaderboard` reads gross `daily_loss` from `casino_account` (player-side Blackjack and Dragon Gate deltas only, so wins don't offset it). The bot is just another player here.
- Casino-system P&L lives in the `casino_ledger` row in `economy.db`; `/casino` reads the ledger, `/pocat` reads the bot's wallet. Do not reuse the bot's `user_wallet` as the house ledger. Player wallet and casino ledger commit in one atomic transaction (`apply_round_settlement`).
- Ephemeral vs public boundary: shared social / market / settlement events are public embeds with scheduled cleanup; personal state and validation / permission failures are ephemeral.
- Only one faucet exists: `cli.py` grants the cooldown-gated per-message reward. Other cogs must not invent action rewards.
- Anti-inflation levers in `typings/economy.py`: `transfer()` burns `TRANSFER_TAX_BPS` of every `/give` as a permanent sink, casino wagers cap at `MAX_SINGLE_BET` at the shared chokepoints, and the VIP casino multiplier is 1.2x (`apply_vip_blackjack_bonus`). Re-measure before changing.

## Stocks

- Stock state lives in `data/database/stock.db`; cash stays in economy `user_wallet` (settlement calls `_economy.database`). Money/share columns are `StoredInteger` (see Economy); wallet deltas stay integer via `cash_ceil` / `cash_floor`. Tables keying `user_id` also persist `user_name`; UI shows stored names, not Discord IDs.
- `/stock` is a `StockPublicView(OwnedPublicView)`: one public message edited in place, opener-only, self-deletes after `STOCK_ACTION_TIMEOUT_SECONDS` (180s) idle. Shared with fishing via `utils/owned_message_views.py`.
- `stock_profile` is the source of truth for virtual companies; maintain it offline via direct DB edits. Do not seed companies from runtime code and do not add schema migrations.
- `float_shares` caps aggregate long exposure and short borrow capacity; opening new long also clamps each user to `STOCK_INDIVIDUAL_OWNERSHIP_CAP_BPS` (49%) of float. Risk-reducing flow (sell long, cover short) is always allowed.
- `stock_news` refreshes at most once per `news_cadence_hours` per symbol; `StockNewsAI` (`_stock/news.py`) may use the LLM, but helpers must fall back to the deterministic `STOCK_NEWS_*_FALLBACK_TEMPLATES` and never block settlement on LLM failure.
- News sentiment is a one-shot impulse applied at its firing tick boundary (`_news_impulse_by_boundary`), never a decayed drift over later ticks; the decayed value (`decay_news_sentiment`) only feeds the AI news prompt as ambient context.
- Market pressure uses windowed decayed order flow over `liquidity_shares` (`pressure_from_order_flow`); do not reintroduce a multi-day aggregate pressure term.
- Global anti-inflation guardrails are constants in `_stock/market.py` (`MARKET_VOLATILITY_SCALE_BPS`, `GLOBAL_MAX_TICK_CHANGE_BPS`, Taiwan-style `apply_daily_price_limit`); re-measure with an offline simulation before changing them.
- Execution price applies order-size slippage bounded by `max_order_impact_bps` (`execution_price_cents` / `order_impact_bps`); `max_tick_change_bps` is the separate per-tick price-move clamp in `calculate_next_price_cents`. Store and display per-leg `price_cents`; never settle large orders at the quote price.
- All settlement goes through `settle_stock_operation`; views never split price / wallet / position reads and writes or import stock ORM models. One operation writes one `stock_operation` plus ordered `stock_trade_leg` rows (`leg_order`); do not net wallet legs.
- Portfolio process cache is invalidated on stock writes; market board and 7D chart PNG caches are `lru_cache`d on every pixel-affecting field, so include every such field in the render key.
- Lazy ticks advance every 5 min on interaction (`STOCK_TICK_SECONDS`); backlogs compress to `MAX_TICKS_PER_INTERACTION` ticks while preserving Asia/Taipei day-rollover boundaries (`tick_boundaries_to_apply`).
- Render overflow as a Pillow PNG via `attachment://...`; shared CJK font and text-anchoring helpers live in `utils/pil_text.py`.

## Games

- Pure rules in `_games/blackjack.py` and `dragon_gate.py`; production wires `random.SystemRandom`, tests inject seeded `random.Random` (clear `round_state.shoe` before monkeypatching `draw_card`). Lobby scaffolding in `lobby.py` keeps `raise NotImplementedError`; do not convert base views to `abc.ABC`.
- **No LLM anywhere**: the former casino `SystemNarrator` banter and bot-player `reason` text were removed; the games carry no flavor text and the cog holds no LLM client. Do not reintroduce a casino narrator or AI bot-player. The dealer / casino is a label, not a Discord identity; no narrator messages.
- **Settlement** is one atomic step after the round resolves: validate or clamp bets up front, then settle through the helpers. Blackjack uses `BlackjackHandState` plus `BlackjackHandSettlement` / `BlackjackPlayerSettlement` (do not reintroduce the legacy single-hand path). Player losses clamp at balance 0; `apply_round_settlement` mirrors only the actual collected debit into `casino_ledger`.
- Blackjack `bet` is a string option, `bet=0` means all in. Split matches Blackjack value (10/J/Q/K interchangeable); no Double after Split; split-hand 21 is not natural. `MAX_BLACKJACK_PLAYERS = 6`.
- 過五關: a 5+ card non-bust hand auto-stands and wins regardless of dealer total. Five-card 21 adds a system-funded 1x bonus that counts as player-side payout but never moves `/casino`. The VIP bonus is `max(0.2x dealer-paid win, 0.2x five-card bonus)`, not the sum.
- The dealer is the casino system, deterministic under H17 (`is_soft_17`), never an LLM. The bot joins every table as a regular player (auto-added when its wallet > 0) and settles like any human; its decisions are deterministic, never LLM: `kelly_bet` (fractional-Kelly, edge from the channel shoe's Hi-Lo true count), the hole-aware `recommended_action` via `choose_bot_action`, count-based insurance (take iff P(ten) > 1/3). The bot never sees the dealer hole card. Re-measure the Kelly / edge constants offline before changing.
- The shoe (default 4 decks) is persistent per channel via `shoe.py::BlackjackShoeStore` (in-memory, rebuilt past `RESHUFFLE_THRESHOLD_CARDS`) so card counting has signal.
- **View UX**: action buttons are presence-based (invalid controls removed, not disabled). Peek is a two-stage animation (`_animate_peek_locked`: hidden hole then revealed, no caption); do not collapse it into a single edit. `BlackjackView._finalize_locked` disables buttons and calls `self.stop()` before settlement; the final edit removes controls with `view=None`. 180s idle timeouts; terminal public messages schedule deletion via `utils.message_cleanup` (no cog-local delete loops). The only background task is off-critical-path round-history persistence (`_record_history_later`).
- One embed per seat: `[dealer_seat, *player_seat_embeds]` (`build_in_progress_embeds` / `build_final_embeds`). Markdown headings render reliably only inside `embed.description`. Player seats set the avatar thumbnail; the dealer seat omits it. Dealer seat color follows the casino-vs-table outcome: any player win red, all-loss green, all-push yellow.
- **Dragon Gate** is backed by the shared `jackpot_pool` row `game_id="dragon_gate"` in `economy.db`, not the casino ledger. Ante, losses, wins, and refunds settle through jackpot helpers; losses clamp at 0; on leave or timeout a positive running delta refunds into the pool unless a whole-pool win already cleared it.

## Fishing

- `/games fishing` is a single-player, LLM-free money sink under `cogs/_fishing/`; state lives in `games.db` through the fishing-owned engine, wallet cash stays in economy. One public message edited in place, opener-only, self-deletes after 180s idle. Stock and fishing share `utils/owned_message_views.py`: subclass `OwnedPublicView` and edit through `edit_owned_public_message(...)`.
- `fish_grade_config`, `fish_species`, and `fishing_gear` are the tunable source of truth, seeded offline; `defaults.py::build_default_catalog` is the one catalog definition.
- `catch.py` is pure and RNG-injected. Luck is the additive clamped rod+bait `rarity_shift_bps`; `compose_grade_weights` never moves the most common grade.
- Net-deflationary by design: every rod+bait combo must stay a sink (each cast's EV is below bait + amortized rod cost), catches cap at `FISHING_MAX_SINGLE_CATCH`; the seeded EV regression in `tests/test_fishing_catch.py` guards this after retuning. Purchases burn via `apply_ordered_wallet_deltas`; payouts credit via `credit_with_repayment`.
- Cross-DB ordering: `purchase_gear` debits the wallet first then grants gear (refund on grant failure); `settle_cast` writes games.db first then credits economy.db, returning `CastStatus.PAYOUT_DEFERRED` on payout failure (never rolled back). Settlement only via `settle_cast` / `purchase_gear`; views never import ORM models. `catch_log` denormalizes species fields so catalog retuning never rewrites history.

## Memory

- File-backed per-user markdown under `data/memories/<user_id>/`, keyed by user id (cross-server, shared with DMs). Tiers: `main.md` (small distilled hot tier, read on demand), `detail.md` (append-only cold evidence, hard-capped at `DETAIL_FILE_MAX_BYTES`, oldest-trimmed), `raw.md` (phase-1 staging), `main.bak.md` (one-generation recovery), `tone.md` (short always-read tone note, see the tone bullet). Layout/trim rationale in `store.py`'s docstring + `constants.py`. Do not move it into SQLite or add legacy-layout fallbacks.
- **Cross-server privacy (source scoping)**: the file stays one-per-user, but every `main.md` bullet outside `## 使用者輪廓` ends with a `[src:...]` tag (`[src:*]` anywhere-safe, `[src:<guild_id>[,...]]` / `[src:dm]` locked to its source, `[src:legacy]` pre-migration unknown source). Write side: phase-1 emits `sharing: global|source_only` per observation (deterministic tighten-only gate in `_sanitize_observation`: `recent_context` or an OTHER-person token forces `source_only` — the target's own id is exempt so a quoted author prefix can't lock a global fact, and the scan runs on PRE-trim text so a token past the truncation can't dodge it); the conversation source rides as the subject's second line (`subject_source_line` / `parse_subject_source`, surviving the `memory_job` round-trip) and is CODE-stamped into raw observation bodies (`- source:` / `- sharing:`), never LLM-echoed; PHASE2 carries both into the bullet tags (copy verbatim, widen-only). Read side: `filter_memory_for_context` (`memory_tool.py`) is the enforcement point next to the allowlist — before injection it drops locked bullets whose source ≠ current guild (owner's own 1:1 DM sees everything; group DM fail-closes to `[src:*]`; untagged/malformed → treated as legacy → dropped; every indented line shares its column-0 parent's fate, so a nested sub-bullet can't outlive a filtered parent) and strips all tags so the model never sees provenance; applied in `resolve_user_memories` (`_participant_memory_fallback` delegates to it). `## 使用者輪廓` is the one prompt-enforced (untaggable) spot: PHASE2 keeps it `[src:*]`-safe, and the filter honors the passthrough only when the file carries ≥1 well-formed tag — an untagged file (failed migration) has no such contract and its profile fails closed too. Dedupe is `(normalized_key, source)` (`filter_duplicate_observations`) so a fact restated in another guild can re-enter raw and widen its tag. `_build_runtime_instructions` injects the current location (`REQUEST_LOCATION_CONTEXT_PROMPT`, guild id ONLY — the owner-controlled guild name must never enter developer-authority instructions); the REPLY_PROMPT rule "never volunteer where a fact was learned" is only the backstop. Server memory needs none of this (per-guild by construction); server-flavor observations hardcode `sharing="global"` and render no source lines. One-shot `scripts/migrate_memory_privacy.py` re-tags pre-existing files (`[src:*]`/`[src:legacy]`, deterministic `tag_untagged_bullets` backstop) and seeds tone.md.
- **Two-phase write** (`cogs/_memory/`, fire-and-forget; must never add reply-path latency; LLM failures silently keep the previous state): phase-1 `MemoryExtractorAI.extract` after each QA reply (per-user in-flight de-dupe replays only the newest skipped turn) + a phase-1.5 evaluator, then deterministic gates (`_validated_draft` / `_is_accepted_observation`, `extraction.py`) reject casual mentions, hypotheticals, bot suggestions, other-user context, weak confidence, non-target observations, and duplicate `normalized_key`s before raw append. The `durability` ladder is `volatile|session|recent|stable|permanent`; the stable-category gate accepts `stable` or `permanent` (the latter marks the never-aged identity facts / enforced directives that route to `## 永久事實`). Phase-2 `consolidate` distills the raw batch + a `read_detail_tail` window into `main.md` once `RAW_CONSOLIDATION_THRESHOLD` accumulates and the cooldown elapses (`RAW_CONSOLIDATION_MAX_BYTES` bypasses cooldown), capped by `MEMORY_GLOBAL_CONCURRENCY`. Thresholds + reasons in `constants.py`. Models: `extract_model` / `memory_evaluator_model` / `memories_model`.
- **Restart-resumable phase-1 inbox** (`cogs/_memory/database.py`, table `memory_job` in `reply.db`): the markdown store stays the source of truth (do NOT move memory *content* into SQLite); this DB only persists the *progress* of the fire-and-forget pipeline so a redeploy mid-extraction is not lost. One row per scope (newest-wins, guarded by an integer `time.time_ns()` token — `database_now()` collides at microseconds and round-trips tz-naive, so it cannot back the guard). `schedule_memory_update` renders the transcript eagerly and `_run_memory_update` writes `status='pending'` at its top; phase-1 terminal marks `done` (transcript nulled); an `extract()` LLM failure marks `failed` (transcript kept), so the restart sweep retries it with no timeout tuning. `ReplyGeneratorCogs.on_ready` (guarded by `_resume_started`, fires every reconnect) re-enqueues non-`done` rows and sweeps every over-threshold scope (`iter_scopes` + `consolidate_if_needed`), including scopes with a resumed job: the scope lock plus the under-lock `_should_consolidate` re-check keep the resumed extraction and the sweep idempotent, so a consolidation a restart interrupted still finishes even when the resumed extraction early-returns. Consolidation itself needs no job rows since `raw.md` is its durable, re-entrant queue. Every DB call is best-effort (`_safe`, swallow + `logfire.warn`), the opposite of `research.py`, so reply.db trouble never breaks the in-memory pipeline.
- **Tone note (`tone.md`)**: persona-independent delivery preferences (the qualities the user wants, never a named persona or the current voice, so a `PERSONA_CHOICES` change never strands it) live in their own per-user tier — cross-server safe by construction, the deliberate exception to source scoping. Written by the SAME phase-2 `consolidate` call as `main.md` (`ConsolidatedMemory.tone_markdown`, no separate updater/cooldown — that was PR #286's rejected shape): `_write_tone_result` accepts only user flavor + a `## 語氣偏好` header, on both the accepted-rewrite and genuine-no-op paths (a main no-op can still carry tone signal); an empty output never deletes the note, and a non-empty MALFORMED tone rejects the whole batch (raw kept for retry) since the accepted main may have moved tone bullets out on the promise they land in the note — PHASE2 is also told never to drop a tone bullet from main unless the same rewrite emits it in `tone_markdown`. The first untagged→tagged rewrite of a pre-migration file is judged by the compaction shrink floor (tone-move + profile privatization legitimately shed >50%), so a failed-migration scope can't freeze. Injected for the message author on EVERY reply with no selection (`render_tone_block`, `ReplyContext.tone_block`, one file read, even on SUMMARY; media persona replies too); `PERSONA_CHOICES` says the note overrides the default voice, delivery only. `TONE_FILE_MAX_BYTES` is a store backstop, shortness lives in the prompt.
- **Anti-injection** (plumbing in `extraction.py`, not `prompts.py`): the transcript renders column-0 `[message <n> | <role>]` markers with indented bodies so user text can't forge a boundary; `sanitize_identity` (`input.py`) neutralizes `[id:`-lookalikes in names; `redact_secrets` (`_SECRET_PATTERNS`, shape-specific, deliberately no bare-hex rule so git SHAs survive) runs on the transcript and every model output. Prompts treat conversation content as data, never instructions.
- **Read path is two-phase on purpose** (latency / cost / provider-neutral; LiteLLM drops grounding when a function tool mixes with the built-in tools, opt-out Preview-only — see Responses API Gotchas). Phase 1 (`_select_user_memories`) offers only the `get_user_memory` tool so the model picks whose memory to read; phase 2 injects the selected memory as a low-authority `role=assistant` block (`render_memory_context_block`). The allowlist (`build_memory_allowlist`: conversation authors + `.mentions`, minus the bot) is the per-user permission boundary, enforced server-side by `resolve_user_memories` before any `main.md` read; the per-bullet source filter (see the cross-server privacy bullet) then scopes what that read may contain, so `resolve_user_memories` takes a `MemoryReadContext`. `widen_allowlist_with_aliases` adds absent members as callable ids only in public channels (`include_absent=_source_channel_is_public`) since the nickname table is public but the personal memory it would unlock is not. When-to-call rule in `MEMORY_SELECT_PROMPT`. SUMMARY/IMAGE run no selection. Pass a memory-free, target-centered `message_list` (`target_centered_memory_messages`) to extraction so it never re-ingests stored memory.
- `main.md` starts with `v1` and has no code-side size clamp: past `MAIN_COMPACTION_TRIGGER_CHARS` the next consolidation compacts toward `MAIN_COMPACTION_TARGET_CHARS` by LLM summarization, never truncation. Rewrite guards: the shared LLM envelope refuses an `incomplete` status and `_rewrite_shrank_too_much` rejects drastic surprise shrinks (keeping raw for retry) so a lossy rewrite never overwrites good memory or its backup. The consolidation input carries a `today:` line for aging. Stable memory has three retention tiers: permanent items (immutable identity facts + enforced standing directives) live undated in `## 永久事實` and never age; mutable durable traits carry a `[~YYYY-MM]` last-confirmed tag and age by *displacement* — a bullet is dropped once its month falls more than `STABLE_FRESHNESS_WINDOW_DAYS` behind the freshest mutable bullet `latest` (anchored on newest activity, NOT on `today`, so a quiet scope forgets nothing and a busy one self-trims; dropped traits self-heal from raw); `## 近期脈絡` keeps its `[YYYY-MM-DD]` ~30-day TTL. The two date shapes (`[~YYYY-MM]` vs `[YYYY-MM-DD]`) are kept disjoint on purpose. Memory is always on (no kill-switch); `memory_enabled` still gates per call.
- `regenerate_main_memory` (`pipeline.py`) is the shared pure-evidence rebuild for `/memory regenerate` (default `memories_model`, own `MEMORY_REGENERATION_COOLDOWN_SECONDS`) and `scripts/regen_memories.py`: rebuilds from the detail tail + unconsumed raw only, never the existing main (`consolidate(existing_main="", compact=True)`), `_rewrite_shrank_too_much` skipped (no prior size). `/memory regenerate` dispatches a de-duped background task via `schedule_memory_regeneration` and replies immediately.
- All writes serialize under `scope_lock(scope)`; main writes are atomic with a `main.bak.md` backup first; in-flight updates check `cleared_since` so a clear isn't resurrected. Only `main.md` carries a single-line author identity (after the `v1` header) for human inspection; read helpers strip it so injection, consolidation, and `/memory show` never see it. Raw/detail headers are timestamp-only. Tests use the `memory_isolated_dir` fixture.
- `store.py` / `pipeline.py` are keyed by an opaque `scope: str` (the subdir under `data/memories/` and the registry key); `user_scope(user_id)` / `server_scope(bot_id, server_id)` build it, and one store + one pipeline serve both flavors (shared `MEMORY_GLOBAL_CONCURRENCY`, per-scope de-dupe + cooldowns). `MemoryExtractorAI` carries its phase prompts as fields; `extract(subject, ...)` takes a per-call `target_user_id:` / `target_server_id:` subject.
- **Per-server (community) memory** at `data/memories/<bot_id>/<server_id>/` via the same engine with `server_prompts.py` (sections 伺服器輪廓 / 社群文化 / 常見話題 / 重要事實 / 成員稱呼 / 近期脈絡; community-level only, never an individual's personal facts). `gen_reply` reads the guild's `main.md` once (`_read_server_memory`) and injects `render_server_memory_block` into both the selection request and the answer (zero extra latency), and schedules a server-scope update after each guild QA or SUMMARY. `## 成員稱呼` is a member nickname-to-id table, the sole carve-out from the no-individuals rule, recorded as `stable_fact` with `durability=permanent` so the freshness pass never ages it; the other server stable sections follow the same `[~YYYY-MM]` displacement aging as user memory. DMs skip both; SUMMARY skips user memory but still records server memory. The write side is gated by `_source_channel_is_public` so content a channel's `@everyone` can't view never feeds the member-readable server memory.
- User-facing commands are read/rebuild only: `/memory show`, `/memory regenerate`, `/memory server show`. There is no `/memory clear`; the store-level `clear_memory` / `mark_cleared` / `cleared_since` primitives stay for the in-flight abort guards and offline maintenance. When clear is exercised, the in-flight and deferred abort branches mark the scope's `memory_job` row `done` so a restart never resumes a cleared turn; an offline memory wipe (files removed while the bot is down) must also drop that scope's `memory_job` row by direct DB edit, or the restart sweep resurrects it (the privacy-redesign deploy did exactly this: delete user-scope `raw.md`/`detail.md`/`main.bak.md`, `DELETE FROM memory_job WHERE flavor='user'`, then run the migration script).

## Conventions

- Environment-backed config classes use `pydantic_settings.BaseSettings` with explicit `validation_alias=AliasChoices("ENV_NAME")`; `.env` is loaded at import time.
- Pure shared result types, enums, and constants live under `typings/` when they don't depend on cogs or utils.
- Use `pydantic.BaseModel` for structured data and type-managed payloads, including service/helper classes. No `dataclass`; no loose `object` / `Any`. Keep `Field(description=..., examples=...)` on every model; a required field uses an explicit ellipsis: `Field(..., description=...)`.
- Ruff is formatter and linter; mypy runs in pre-commit. Use narrow `# noqa: <rule>` comments with a reason.
- Keyword arguments are required for normal calls, including single-argument ones; do not add bare `*`. Allowed positional idioms: `len(x)`, `str(x)`, `Path("x")`, exception constructors, variadic collectors, `logfire.info("message")`.
- Avoid intermediate one-level aliases such as `usage = responses.usage`.
- Import functions by name and call them unqualified (`from pkg.mod import do_thing`); a module-namespace import is a last resort with a reason.
- LLM latency matters; do not add extra LLM calls for cosmetic improvements.
