# CLAUDE.md

Project-specific guidance for Claude Code when working in this repository. This file is dense and AI-agent-facing on purpose: it is an index of pointers, cross-cutting invariants, and non-obvious pitfalls, not a re-explanation of what the code already documents. When a fact already lives in a code comment or docstring, this file keeps only the pointer plus the load-bearing "why". `AGENTS.md` and `GEMINI.md` are symlinks to this file, so edit `CLAUDE.md` only.

## Commands

All tooling runs through `uv`.

```bash
uv run discordbot                # run the bot
uv run pytest                    # tests, coverage gate: 80% (--cov-fail-under)
uv run pre-commit run -a         # canonical pre-push check (= make fmt)
make gen-docs                    # regenerate docs/ from sources
```

When debugging, check `./data/logs` for the full runtime logs.

## Runtime Shape

- `src/discordbot/cli.py` defines `DiscordBot(commands.Bot)`; intents are `Intents.all()` minus `members` and `presences`.
- Cogs load synchronously in `__init__` before the gateway connects. Every cog module exposes a sync `def setup(bot)` and adds the cog with `override=True`; an `async def setup` breaks the first command sync (nextcord schedules it without awaiting). See `_load_cogs_sync` in `cli.py`.
- Helper packages live in sibling `_<cog>/` dirs so the `cogs/*.py` loader glob does not pick them up as cogs.
- Bot presence is hardcoded in `status_task`; there is no DB-backed rotation.
- Add common command errors in `DiscordBot.on_command_error`, not in each cog.

## Cog Rules

- One `commands.Cog` subclass plus a sync `setup` per module. Cogs do not import peer cogs; reach across via the bot instance, shared `typings/` / `utils/`, or cog-private `_<cog>/` packages.
- User-facing slash commands need localized name and description for English, Traditional Chinese (`Locale.zh_TW`), and Japanese (`Locale.ja`).
- Any user-visible command or behavior change must update `cogs/help.py` in the same change and keep `tests/test_help.py` green (it asserts every non-help slash command name appears in every localized help body).
- For multiple embeds needing aligned widths, use `utils.discord_embeds.embed_spacer_payload(..., target=)` so an edit reuses the uploaded spacer; re-uploading it every edit trips Discord error 400009 on rapidly edited messages (see its docstring).
- Content past embed limits is paginated or rendered to a PNG via `attachment://...`; never truncate silently.
- Never hardcode a Discord attachment-size ceiling: read the destination's real limit (`message.guild.filesize_limit` / `interaction.guild.filesize_limit`) so a boosted server's 50/100MB is honored, falling back to Discord's non-Nitro base of 10MB only when there is no guild (e.g. a DM, which exposes no per-DM limit to query). `filesize_limit` is nextcord's boost-tier table lookup keyed on the API's `premium_tier`, not a live API field, so it currently over-reports 25MB for unboosted tier 0/1 (Discord lowered the real base to 10MB in 2024); that self-corrects when nextcord refreshes its table, so the guild path keeps trusting it. Reserve a small margin for the multipart envelope when the upload rides with embeds (`parse_threads.py` holds back 1 MiB). Call sites: `parse_threads.py`, `video.py`, `_gen_reply/streaming.py` (voice clip).

## AI Pipeline

- Runtime LLM backend is LiteLLM Proxy behind `OPENAI_BASE_URL`; all runtime conversations use `AsyncOpenAI` and the OpenAI Responses API. Do not switch back to Chat Completions and do not import provider-native SDKs (`google-genai`, `anthropic`) into request paths. The only carve-outs are the Gemini Files API upload side-channel (`utils.llm.create_gemini_client`, used in `_gen_reply/input.py` so a file can be polled to ACTIVE before use; the answer still references it through the proxy) and `scripts/prompt_dev.py`. The proxy indirection is the point: it keeps the whole project on one OpenAI SDK while the backend stays mostly Gemini and any model is hot-swappable (the Gemini Files API is the lone non-proxied path).
- LiteLLM's published docs lag its code, so most proxy-side behavior questions (request translation, `extra_body` handling, tool / grounding quirks) are faster to answer from source than from docs: `git clone https://github.com/BerriAI/litellm`, check out its latest tag, and read the actual code rather than trusting the docs.
- Runtime model strings live in `RuntimeModelCatalog` (`typings/models.py`); each property's docstring carries its callers and tier rationale (`route_model` is flash-lite, `tool_model` is flash for nickname-to-id matching, `slow_model` keeps a same-model peak/off-peak split on purpose). Provider selection is via the model string plus `extra_body`.
- LLM SDK objects are named `responses`; loop items are named `response`.
- Author runtime prompts in English: system/developer instructions, tool descriptions, and routing/memory/voice prompts in `prompts.py` / `server_prompts.py` (and the TTS `TTS_STYLE_DIRECTIVE`) are written in English. Reply *content* is localized at runtime by the in-prompt "follow the user's language" rule, not by writing the prompt itself in another language. The only deliberate exception is the few Chinese-specific behavior notes that are load-bearing for behavior (e.g. the 轉帳 caution) in `COMMON_PROMPT`; even the persona voice in `PERSONA_CHOICES` is authored in English now (only the bot's own name 破貓 stays as a proper noun).
- AI progress is reactions on the user's message via `utils.reactions.ReactionStatusChain` (ordered, fire-and-forget, flushed at pipeline end); QA shows 💭. There are no separate status messages: while the answer model thinks, `ResponseStreamer` previews the reasoning summary as `-#` subtext in the single reply, replaced by the first content delta. Discord edits run on a snapshot editor task that shuts down via a stop-event, not cancellation, so an in-flight write always lands before the final edit (else it can orphan the reply). The preview/final reply (`_reply_or_send`) degrades from `message.reply` to `channel.send` when the source was deleted (Discord 50035 / `NotFound`), so deleting the source mid-reply no longer wastes the whole pipeline. See `_gen_reply/streaming.py`.
- Spoken replies (`_gen_reply/voice.py`): the answer model decides per reply whether the message should also be said aloud by appending the `VOICE_MARKER` (`</need-voice>`) as the final line; the decision stays inside the answer model on purpose so the written text and the spoken clip stay coherent (a separate post-hoc classifier would not). The switch lives in `COMMON_PROMPT`, so both the QA and SUMMARY answers can use it. `ResponseStreamer` strips the marker from the preview and the final text (`strip_voice_marker` / `strip_partial_voice_marker`), and when present calls `VoiceSynthesizer` after the text is already on screen (zero added latency to the text), then edits the WAV onto the same reply via `Message.edit(file=...)`. It rides alongside the existing grounding tools without a new function tool, so it never risks the grounding silent-drop. Best-effort: any TTS failure, or a clip past the guild's real upload limit (`message.guild.filesize_limit`, DM fallback 25MB, checked at the `streaming.py` attach site) leaves a text-only reply; there is deliberately no spoken-length cap (the answer model decides how much to say, though `COMMON_PROMPT` still nudges it short) and `VOICE_TIMEOUT_SECONDS` is generous so a longer clip has room to render. Enabled on the QA and SUMMARY routes (`allow_voice`) and gated by `LLMConfig.voice_reply_enabled` (the runtime kill-switch: off still strips the marker but synthesizes nothing); IMAGE/VIDEO never speak. TTS quirks (proxy ignores `instructions`, 500s on `response_format` so style rides in the input prefix and output is WAV, single fixed `TTS_VOICE`) are in the module docstring.
- Attachment ingestion is gated by `get_supported_modalities` for the slow model (`MessageInputBuilder._supported_sources`); unsupported attachments are dropped before any LLM call. Rendering is provider-aware on `_answer_model_is_gemini` (`"gemini" in slow_model.name`): a non-Gemini answer model cannot resolve a Gemini Files URI (the proxy mistranslates it), so its attachments are inlined per type by `_inline_file_part` (image base64, PDF base64, UTF-8 text, else dropped; needs no `GEMINI_API_KEY`). Gemini stays the default because it also ingests video and hits the inline cap sooner. The Gemini path uploads every supported attachment to the Files API and references the full `uri` as an `input_file` `file_id`, never base64 (avoids the per-part `inline_data` cap); the bare `files/<id>` name fails the proxy's mime lookup. The rest is in `_gen_reply/input.py` docstrings: `_upload_file`'s 15s ACTIVE-poll and the deprecated-`uploaded`-status / "not in an ACTIVE state" 400 pitfall, `PendingUpload` / `_resolve_file_upload` re-poll-and-adopt with a lazy `load_data` so adoption never re-downloads, and the per-message LRU cache keyed on message id, edit time, and `collect_attachment_sources` ids. Images downscale to 3072px longest edge via `utils.images.shrink_image_bytes` first. The IMAGE route is the exception: `_handle_image_reply` needs raw pixels for `images.edit`, so it loads bytes via `get_image_source_bytes`, not Files-API handles. Media fetch + upload is bounded by `MEDIA_CONCURRENCY` (`_media_semaphore`, shared on the singleton builder) so concurrent pipelines do not starve each other; URL image fetches use a 10s timeout (distinct from the 15s ACTIVE-poll); a source whose fetch fails is skipped for `DEAD_SOURCE_TTL` so an expired-CDN attachment in scrollback is not re-fetched and re-warned every reply.
- The IMAGE and VIDEO routes refine the prompt before generation: `_handle_image_reply` / `_handle_video_reply` first run a best-effort director (`_refine_generation_prompt` on `prompt_model`, flash+high) that expands the user's short request into one rich, self-contained generation prompt, calling the built-in `googleSearch` / `urlContext` tools to look up named subjects (no function tool mixed in, so no grounding silent-drop). `IMAGE_PROMPT` and `VIDEO_PROMPT` (`_gen_reply/prompts.py`) drive it; `VIDEO_PROMPT` is motion / camera / pacing oriented. Image-edit passes its already-loaded `get_image_source_bytes` bytes as input images so the draft is grounded with no re-download; video is text-only. Any director error or empty output falls back to the raw prompt (it must never raise into the pipeline's outer error path), with the existing Chinese default as the last-ditch fallback. It sits serially before generation on the critical path, so `effort` is the latency lever; logged as `gen_reply prompt refine`. `DESCRIPTION_PROMPT` is the separate post-generation caption (`fast_model`).
- For Discord embeds prefer `media.discordapp.net` `proxy_url` over origin URLs (CDN links expire).
- Reference/current and history are rendered twice from one walk: text-only twins (`_get_reference_and_current_text_only`, `process_single_message_text_only` emitting `[attachment: kind]` markers, no upload) feed routing and memory selection, while the uploaded-`file_id` render (`_get_reference_and_current`) runs as a background task only the answer awaits, so the route never blocks on the Files API and each upload overlaps the route + selection latency. `RenderedHistory` carries both renders plus the raw side feeding the allowlist from one history fetch; `_prepare_reply_context` -> `ReplyContext` builds history + server memory + memory selection speculatively, concurrent with the route and the parallel effort grade. Non-QA routes cancel it (IMAGE/VIDEO also cancel the upload task); SUMMARY rebuilds at `history_limit=100`, `memory_enabled=False`, reusing the shared parts. The route is split into two parallel calls under one `route_done` gate: `_route_classify` returns `RouteClassification` (the `decision`; the only critical-path call that gates dispatch) and `_grade_effort` returns `EffortGrade` concurrently; QA and SUMMARY override the slow model's effort with the grade and the usage footer shows `model (effort)`, while IMAGE/VIDEO cancel the effort task since they never consume it. Memory selection and effort grading share the `route_done`-gated `_await_gated` helper (each runs unbounded while the route is in flight, then gets only a grace once the route returns: `MEMORY_SELECT_GRACE_SECONDS` / `EFFORT_GRACE_SECONDS`, not a fixed timeout); effort grading falls back to `high` on timeout or error. Memory selection on timeout or error falls back to the memory of the message author plus any reply-reference authors (`_participant_memory_fallback`, allowlist-gated so a reply to the bot reads nothing for it) rather than dropping memory entirely; a completed selection that deliberately picked nobody still injects nothing. `_handle_video_reply` is bounded by `VIDEO_GENERATION_TIMEOUT_SECONDS`. Logfire spans are `gen_reply pipeline` / `gen_reply route` / `gen_reply effort` / `gen_reply context build` / `gen_reply memory selection` / `gen_reply answer`, plus first-reasoning-delta and first-content-delta logs in `streaming.py`.
- When the current message carries a Threads URL (`THREADS_URL_RE`, now in `utils/threads.py`, the single definition the `parse_threads` cog also imports), `gen_reply` self-parses it metadata-only (`ThreadsDownloader.parse_metadata`, no disk download) via `build_threads_context_messages` (`_parse_threads/builder.py`) on a `route_done`-gated `_await_gated` task (`THREADS_GRACE_SECONDS`), independent of the `parse_threads` cog. It injects the post's text metadata plus its image/video as URL parts (`input_image` `image_url` / `input_file` `file_url` — raw signed Threads CDN URLs the proxy fetches server-side, verified end-to-end on Gemini; no download, no Discord proxy) into `ReplyContext.threads_block` just before the current message on the QA route. Provider-aware: a non-Gemini answer model gets the URLs as text instead. Text metadata is injected unconditionally (a private/deleted post yields a "could not read it" notice, never a hallucinated body), so the model never falls back to "I cannot open this link" — the failure mode of the reverted #294, which raced the `parse_threads` cog's download+post through a relay. Only the first URL is parsed; IMAGE/VIDEO cancel the task; a URL-bearing SUMMARY is rerouted to QA by `_route_classify`. Span `gen_reply threads context`.

### Responses API Gotchas

- Strict role and content part pairing: `user` / `system` / `developer` use `input_*` parts; `assistant` uses `output_text` or `refusal`. So a message with attachments falls back to `role=user` (see `_assemble_input_message`). Prefer string-content shorthand when there are no attachments.
- The top-level `instructions` parameter has `developer` authority and outranks everything in `input`, but does not persist across turns. Put behavior rules there (or in `COMMON_PROMPT`), never in an input-side wrapper.
- Separator messages use `role=system`, not `developer`, for Gemini and Claude compatibility through LiteLLM.
- When a custom function tool rides with the built-in `googleSearch` / `urlContext` in one Gemini request, LiteLLM silently drops the grounding (search annotations go to zero, no 400). So when mixing tool kinds, verify the grounding still fires, not just that the call succeeds. The Gemini 3 opt-out that lifts it (`include_server_side_tool_invocations: true` flat in `extra_body`) is Preview-only, so memory retrieval stays two-phase on purpose: the selection request carries only the function tool and the answer carries only the built-in tools. Rationale is in `_prepare_reply_context` and the Memory section.
- The code-execution built-in tool is intentionally omitted from `ModelSettings.tools`: providers validate every uploaded file part against its narrow MIME allowlist and 400 the whole request on video / audio / GIF-as-video, so it cannot coexist with the Files-API attachment path. The search/url tools have no such limit.
- Gemini prepends leading newlines to the first reasoning and content delta; `_on_reasoning_delta` / `_on_content_delta` strip them. Gemini thought summaries only flow through the Responses API.
- `client.images.edit(image=...)` needs raw `bytes`, not image-reference dicts.
- Per-token pricing comes from `utils.model_pricing` and its cached LiteLLM JSON; do not hardcode rates.

## Economy

- DB files under `data/database/`: `messages.db` (log_msg), `economy.db` (per-user 虛擬歡樂豆, jackpot pools, casino ledger), `games.db` (Blackjack history, fishing state, cleanup targets), `stock.db`. `_games` / `_fishing` / `message_cleanup` each keep their own engine on the shared `games.db`; intentional.
- `_economy/database.py` owns the module-level `_engine` (its docstring has the pool-leak + test-monkeypatch reason); do not move it to `cached_property`.
- `UserAccount` has no `guild_id`: identity, VIP, check-ins, and flags are cross-server. Money columns are `StoredInteger` decimal text parsed to Python `int` to dodge SQLite's 64-bit ceiling; balance in `user_wallet`, loans in `loan_proposal` / `loan_contract`, daily casino counters in `casino_account`.
- Invariant: every applied positive delta bumps `total_earned`, every negative bumps `total_spent`, so `total_earned - total_spent == balance`. There is no transaction table.
- Money inputs are string `SlashOption`s parsed via `utils.amount_parsing.parse_decimal_amount`, wrapped per domain (`_parse_positive_amount` / `_parse_collect_amount`, `parse_wager_amount`, `parse_bait_quantity`); malformed text returns `build_invalid_amount_embed` ephemerally before any mutation. Reuse this for any numeric input that can exceed Discord's int cap.
- `UserAccount.avatar_url` is a last-seen cache via `utils.avatars.guild_avatar_url`; do not backfill existing URLs.
- `credit_with_repayment` is the single income facade (message reward, casino payout); passive income and `/give` recipients do not auto-repay debt.
- Loans: personal loans debit the lender on acceptance; central-bank loans mint on approval and burn on repayment / collection (no lender credit). Proposals expire after `LOAN_PROPOSAL_TIMEOUT_SECONDS` (180s). Acceptance prepays `MIN_INTEREST_DAYS` of interest, so do not zero `interest_due` at contract creation.
- `is_central_banker` (set offline by direct DB write) is separate from Discord admin; keep `ECONOMY_ALLOW_CENTRAL_BANK_SELF_APPROVAL` unset or `false` in production.
- Admin tweaks go through `adjust_balance(..., allow_negative=...)`, never casino settlement helpers (it skips loans and daily casino counters).
- `/leaderboard` sorts `StoredInteger` text DB-side with `LIMIT` before rows reach Python (`top_n`); `hide_from_leaderboard` accounts are omitted; write paths must call `invalidate_economy_leaderboard_cache()`. `/loss_leaderboard` reads gross `daily_loss` from `casino_account` (player-side Blackjack and Dragon Gate deltas only, so wins do not offset it). The bot is just another player here.
- Casino-system P&L lives in the `casino_ledger` row in `economy.db`; `/casino` reads the ledger, `/pocat` reads the bot's wallet. Do not reuse the bot's `user_wallet` as the house ledger. Player wallet and casino ledger commit in one atomic transaction (`apply_round_settlement`).
- Ephemeral vs public boundary: shared social / market / settlement events are public embeds with scheduled cleanup; personal state and validation / permission failures are ephemeral.
- Only one faucet exists: `cli.py` grants the cooldown-gated per-message reward. Other cogs must not invent action rewards.
- Anti-inflation levers in `typings/economy.py`: `transfer()` burns `TRANSFER_TAX_BPS` of every `/give` as a permanent sink, casino wagers cap at `MAX_SINGLE_BET` at the shared chokepoints, and the VIP casino multiplier is 1.2x (`apply_vip_blackjack_bonus`). Re-measure before changing.

## Stocks

- Stock state lives in `data/database/stock.db`; cash stays in economy `user_wallet` (settlement calls `_economy.database`).
- `/stock` is a `StockPublicView(OwnedPublicView)`: one public message edited in place, opener-only, self-deletes after `STOCK_ACTION_TIMEOUT_SECONDS` (180s) idle. Shared with fishing via `utils/owned_message_views.py`.
- `stock_profile` is the source of truth for virtual companies; maintain it offline via direct DB edits. Do not seed companies from runtime code and do not add schema migrations.
- `float_shares` caps aggregate long exposure and short borrow capacity; opening new long also clamps each user to `STOCK_INDIVIDUAL_OWNERSHIP_CAP_BPS` (49%) of float. Risk-reducing flow (sell long, cover short) is always allowed.
- `stock_news` refreshes at most once per `news_cadence_hours` per symbol; `StockNewsAI` (`_stock/news.py`, prompts in `_stock/prompts.py`) may use the LLM, but helpers must fall back to the deterministic `STOCK_NEWS_*_FALLBACK_TEMPLATES` and never block settlement on LLM failure.
- News sentiment is a one-shot impulse applied at its firing tick boundary (`_news_impulse_by_boundary`), never a decayed drift over later ticks; the decayed value (`decay_news_sentiment`) only feeds the AI news prompt as ambient context.
- Market pressure uses windowed decayed order flow over `liquidity_shares` (`pressure_from_order_flow`); do not reintroduce a multi-day aggregate pressure term.
- Global anti-inflation guardrails are constants in `_stock/market.py` (`MARKET_VOLATILITY_SCALE_BPS`, `GLOBAL_MAX_TICK_CHANGE_BPS`, Taiwan-style `apply_daily_price_limit`); re-measure with an offline simulation before changing them.
- Execution price applies order-size slippage bounded by the profile's `max_order_impact_bps` (`execution_price_cents` / `order_impact_bps`); `max_tick_change_bps` is the separate per-tick price-move clamp in `calculate_next_price_cents`. Store and display per-leg `price_cents`; never settle large orders at the quote price.
- All settlement goes through `settle_stock_operation`; views never split price / wallet / position reads and writes or import stock ORM models. One operation writes one `stock_operation` plus ordered `stock_trade_leg` rows (`leg_order`); do not net wallet legs.
- Money and share columns are `StoredInteger` decimal text parsed to `int`; wallet deltas stay integer via `cash_ceil` / `cash_floor`. Tables keying `user_id` also persist `user_name`; UI shows stored names, not Discord IDs.
- Portfolio process cache is invalidated on stock writes; market board and 7D chart PNG caches are `lru_cache`d on every pixel-affecting field, so include every such field in the render key.
- Lazy ticks advance every 5 min on interaction (`STOCK_TICK_SECONDS`); backlogs compress to `MAX_TICKS_PER_INTERACTION` ticks while preserving Asia/Taipei day-rollover boundaries (`tick_boundaries_to_apply`).
- Render overflow as a Pillow PNG via `attachment://...`; shared CJK font and text-anchoring helpers live in `utils/pil_text.py`.

## Games

- Pure rules in `cogs/_games/blackjack.py` and `dragon_gate.py`; production wires `random.SystemRandom`, tests inject seeded `random.Random` (clear `round_state.shoe` before monkeypatching `draw_card`).
- Lobby scaffolding in `lobby.py` keeps `raise NotImplementedError`; do not convert base views to `abc.ABC`.
- Casino settlement is one atomic step after the round resolves; validate or clamp bets up front, then settle through the helpers.
- Blackjack `bet` is a string option, `bet=0` means all in. Split matches Blackjack value (10/J/Q/K interchangeable); no Double after Split; split-hand 21 is not natural. `MAX_BLACKJACK_PLAYERS = 6`.
- Settlement uses `BlackjackHandState` plus `BlackjackHandSettlement` / `BlackjackPlayerSettlement`; do not reintroduce the legacy single-hand path.
- 過五關: a 5+ card non-bust hand auto-stands and wins regardless of dealer total. Five-card 21 adds a system-funded 1x bonus that counts as player-side payout but never moves `/casino`. The VIP bonus is `max(0.2x dealer-paid win, 0.2x five-card bonus)`, not the sum.
- The dealer is the casino system, deterministic under H17 (`is_soft_17`), never an LLM.
- The bot joins every table as a regular player (auto-added when its wallet > 0) and settles like any human. Its decisions are deterministic, never LLM-chosen: `kelly_bet` (fractional-Kelly, edge from the channel shoe's Hi-Lo true count), the EV engine's hole-aware `recommended_action` via `choose_bot_action`, and count-based insurance (take iff P(ten) > 1/3). Re-measure the Kelly / edge constants with an offline simulation before changing them.
- `BotPlayerAI` only narrates `reason` text off the critical path (deterministic template shows first, the LLM upgrade is a revision-guarded background refresh). The model never sees the dealer hole card or anything derived from it: the per-action EVs shown to it come from `blackjack_ev.py`'s hole-independent marginal pass. Prompts in `cogs/_games/prompts.py` are narration-only; do not turn them back into decision-making or strategy lookup.
- The shoe (default 4 decks) is persistent per channel via `shoe.py::BlackjackShoeStore` (in-memory, rebuilt past `RESHUFFLE_THRESHOLD_CARDS`) so card counting has signal.
- Action buttons are presence-based: invalid controls are removed from the view, not left visible and disabled.
- Player interactions and final result publication must not wait on LLM latency; hit hints and settlement banter are background refreshes.
- Peek is a two-stage animation (`_animate_peek_reveal_bj_locked` / `_animate_peek_no_bj_locked`); do not collapse it into a single edit.
- `BlackjackView._finalize_locked` disables buttons and calls `self.stop()` before settlement; the final edit removes controls with `view=None`.
- Player losses clamp at balance 0; `apply_round_settlement` mirrors only the actual collected debit into `casino_ledger`.
- Dragon Gate is backed by the shared `jackpot_pool` row `game_id="dragon_gate"` in `economy.db`, not the casino ledger. Ante, losses, wins, and refunds settle through jackpot helpers; losses clamp at 0; on leave or timeout a positive running delta refunds into the pool unless a whole-pool win already cleared it.
- Interactive views use 180s idle timeouts; terminal public messages schedule deletion through `utils.message_cleanup`. No cog-local delete loops.
- Game messages send one embed per seat: `[talk, dealer_seat, *player_seat_embeds]` (`build_in_progress_embeds` / `build_final_embeds`). Markdown headings render reliably only inside `embed.description`. Player seats set the avatar thumbnail; the dealer seat omits it. Dealer seat color follows the casino-vs-table outcome: any player win red, all-loss green, all-push yellow.
- The "dealer" is a label, not a Discord identity. `SystemNarrator` lines are sent by the bot account so `log_msg.py` records the speaker correctly.
- `GamesCogs.narrator` and `GamesCogs.bot_player_ai` are `cached_property`s; each cog with an LLM client owns its own client intentionally.

## Fishing

- `/games fishing` is a single-player, LLM-free money sink under `cogs/_fishing/`; state lives in `data/database/games.db` through the fishing-owned engine, wallet cash stays in economy.
- One public message edited in place, opener-only, self-deletes after 180s idle. Stock and fishing share `utils/owned_message_views.py`: subclass `OwnedPublicView` and edit through `edit_owned_public_message(...)` (stock passes its board PNG via `file=`).
- `fish_grade_config`, `fish_species`, and `fishing_gear` are the tunable source of truth, seeded offline; `defaults.py::build_default_catalog` is the one catalog definition.
- `catch.py` is pure and RNG-injected. Luck is the additive clamped rod+bait `rarity_shift_bps`; `compose_grade_weights` never moves the most common grade.
- Net-deflationary by design: every rod+bait combo must stay a sink (each cast's EV is below bait + amortized rod cost), catches cap at `FISHING_MAX_SINGLE_CATCH`; the seeded EV regression in `tests/test_fishing_catch.py` guards this after retuning. Purchases burn via `apply_ordered_wallet_deltas`; payouts credit via `credit_with_repayment`.
- Cross-DB ordering: `purchase_gear` debits the wallet first then grants gear (refund on grant failure); `settle_cast` writes games.db first then credits economy.db, returning `CastStatus.PAYOUT_DEFERRED` on payout failure (never rolled back).
- Settlement only via `settle_cast` / `purchase_gear`; views never import ORM models. `catch_log` denormalizes species fields so catalog retuning never rewrites history.

## Memory

- File-backed per-user markdown under `data/memories/<user_id>/`, keyed by user id (cross-server, shared with DMs). Tiers: `main.md` (small distilled hot tier the reply model reads on demand), `detail.md` (append-only cold evidence, hard-capped at `DETAIL_FILE_MAX_BYTES` with oldest-entry trimming), `raw.md` (phase-1 staging), `main.bak.md` (one-generation recovery). Layout and trim rationale are in `store.py`'s module docstring and `constants.py`. Do not move it into SQLite or add legacy-layout fallbacks.
- Two-phase write pipeline in `cogs/_memory/`: fire-and-forget `schedule_memory_update` runs phase-1 extraction (`MemoryExtractorAI.extract`) after each QA reply (per-user in-flight de-dupe replays only the newest skipped turn), a phase-1.5 evaluator re-reviews, then deterministic gates (`_validated_draft` / `_is_accepted_observation` in `extraction.py`) reject casual mentions, hypotheticals, bot suggestions, other-user context, weak confidence, non-target observations, and duplicate `normalized_key`s before raw append. Phase-2 `consolidate` distills the raw batch plus a `read_detail_tail` window into `main.md` once `RAW_CONSOLIDATION_THRESHOLD` accumulates and `MEMORY_CONSOLIDATION_COOLDOWN_SECONDS` elapses (the `RAW_CONSOLIDATION_MAX_BYTES` trigger bypasses the cooldown), capped by the `MEMORY_GLOBAL_CONCURRENCY` semaphore. Thresholds and their reasons are in `constants.py`. Models: phase-1 `extract_model`, evaluator `memory_evaluator_model`, phase-2 `memories_model`, all in `RuntimeModelCatalog`. Tone/voice preferences are recorded persona-independently (the qualities the user wants, never a named persona or the current bot voice) so a `PERSONA_CHOICES` change never strands a stale persona-bound tone bullet; phase-2 rephrases any existing one. Memory must never add reply-path latency; LLM failures silently keep the previous state.
- Anti-injection invariants (the plumbing is in `extraction.py`, not `prompts.py`): the transcript renders column-0 `[message <n> | <role>]` markers with indented bodies so user text cannot forge a boundary, `sanitize_identity` (`_gen_reply/input.py`) neutralizes `[id:`-lookalikes in names, and `redact_secrets` (`_SECRET_PATTERNS`, shape-specific, deliberately no bare-hex rule so git SHAs survive) runs on the transcript and every model output. Prompts in `prompts.py` / `server_prompts.py` treat conversation content as data, never instructions.
- The read path is two-phase on purpose (latency / cost / provider-neutral; LiteLLM silently drops grounding when a function tool mixes with the built-in search/url tools, and the Gemini 3 opt-out is Preview-only, see the Responses API Gotchas). Phase 1 (`_select_user_memories`) offers only the `get_user_memory` tool (`_gen_reply/memory_tool.py`) so the model picks whose memory to read; phase 2 streams the answer with the built-in tools and injects the selected memory as a low-authority `role=assistant` block (`render_memory_context_block`). The allowlist (`build_memory_allowlist`: conversation authors + `.mentions`, minus the bot) is the permission boundary, enforced server-side by `resolve_user_memories` before any `main.md` read. `widen_allowlist_with_aliases` suffixes present participants with their `## 成員稱呼` row in every guild channel, but only adds absent members as new callable ids in public channels (`include_absent=_source_channel_is_public`) since the nickname table is public while the personal memory it would unlock is not. The when-to-call rule lives in `MEMORY_SELECT_PROMPT` (developer authority), never an input wrapper. SUMMARY and IMAGE run no selection phase. Pass a memory-free, target-centered `message_list` (`target_centered_memory_messages`) to `schedule_memory_update` so extraction never re-ingests stored memory.
- `main.md` starts with `v1` and has no code-side size clamp: past `MAIN_COMPACTION_TRIGGER_CHARS` the next consolidation compacts toward `MAIN_COMPACTION_TARGET_CHARS` by LLM summarization, never code truncation. Rewrite guards: the shared LLM envelope refuses an `incomplete` response status and `_consolidate_locked`'s `_rewrite_shrank_too_much` rejects drastic surprise shrinks, keeping raw for retry so a lossy rewrite never overwrites good memory or its backup. The consolidation input carries a `today:` line so the dated `## 近期脈絡` section can age out. Memory is always on; the `memory_enabled` parameter still gates per call.
- `regenerate_main_memory` (`pipeline.py`) is the shared pure-evidence rebuild for `/memory regenerate` (default `memories_model`, own `MEMORY_REGENERATION_COOLDOWN_SECONDS`) and `scripts/regen_memories.py`. It rebuilds `main.md` from the detail tail window plus unconsumed raw only, never the existing main (`consolidate(existing_main="", compact=True)`), and `_rewrite_shrank_too_much` is intentionally skipped (no prior size). `/memory regenerate` does not run it inline: `schedule_memory_regeneration` dispatches a de-duped background task and replies immediately.
- All writes serialize under `scope_lock(scope)`; main-file writes are atomic with a `main.bak.md` backup first, and in-flight updates check `cleared_since` so a clear is not resurrected. Tests use the `memory_isolated_dir` fixture. Only `main.md` carries a single-line author identity (after the `v1` header) for human inspection; read helpers strip it so injection, consolidation input, and `/memory show` never see it. Raw and detail entry headers are timestamp-only.
- `store.py` and `pipeline.py` are keyed by an opaque `scope: str` that is both the subdir under `data/memories/` and the registry key; `user_scope(user_id)` and `server_scope(bot_id, server_id)` build it, and one store plus one pipeline serve both flavors (shared `MEMORY_GLOBAL_CONCURRENCY`; per-scope de-dupe and cooldowns). `MemoryExtractorAI` carries its phase prompts as fields and `extract(subject, ...)` takes a per-call `target_user_id:` / `target_server_id:` subject, so the same engine drives both flavors.
- Per-server (community) memory lives at `data/memories/<bot_id>/<server_id>/` via the same engine with `_memory/server_prompts.py` (sections 伺服器輪廓 / 社群文化 / 常見話題 / 重要事實 / 成員稱呼 / 近期脈絡, same `v1` contract; community-level only, never an individual's personal facts). `gen_reply` reads the current guild's `main.md` once (`_read_server_memory`) and injects `render_server_memory_block` into both the selection request and the answer (zero extra LLM latency), and schedules a server-scope `schedule_memory_update` after each guild QA or SUMMARY reply. `## 成員稱呼` is a member nickname-to-id table, the sole carve-out from the no-individuals rule, recorded as community vocabulary (`stable_fact`) so the shared gate accepts it. DMs skip both; SUMMARY skips user memory but still records server memory. The write side is gated by `_source_channel_is_public` so content from a channel `@everyone` cannot view never feeds the member-readable server memory.
- User-facing commands are read/rebuild only: `/memory show`, `/memory regenerate` (per-user), and `/memory server show` (guild-only, current guild). There is no `/memory clear`; the store-level `clear_memory` / `mark_cleared` / `cleared_since` primitives stay for the in-flight abort guards and offline maintenance.

## Other Cogs

- `parse_threads.py` watches `threads.net` / `threads.com` URLs (`THREADS_URL_RE`, defined in `utils/threads.py` and shared with `gen_reply`) and uses reactions for status; no extra reward. A reply URL expands the whole reply chain into one message and `_build_embeds` fills the 10-embed cap target-first (its comments have the budget detail). There is no media-count guard on purpose: an oversized carousel degrades to its first images rather than refusing the post; the `total_size` and 4096-char text guards stay. This cog still downloads the target video to disk for its embeds (`parse`, a context manager); `gen_reply` reads the same posts URL-only via the separate `parse_metadata` (no download). The two paths are independent — a mentioned message with a Threads link triggers both.
- `video.py` keeps progress text on the deferred original message, then edits that same message with the final file and source URL.
- Scratch media downloads go to the system temp dir: the cogs pass `tempfile.gettempdir()` to `VideoDownloader` / `ThreadsDownloader` (`utils/downloader.py`, `utils/threads.py`), which require `output_folder` (no default). Files are deleted after delivery. Never put scratch downloads under `data/`.
- `auto_unmute.py` clears the bot's own timeout, finds the moderator from the recent audit-log window, and replies in the last active human channel or the guild system channel (its docstrings have the flow).
- `maplestory.py` + `cogs/_maplestory/`: the `/maplestory` lookup group (monster, equip, scroll, npc, quest, map, item, stats) over local JSON in `./data/maplestory`, maintained offline; a standard cog already covered in `_help/content.py`.
- `log_msg.py` logs human messages and this bot's own replies, never third-party bots; streaming replies converge via UPSERT on `discord_message_id`. It owns the module-level `_sql_engine`; do not move it to a cached property (per-message `cached_property` leaks the pool / dialect / inspector cache, see the comment).

## Config And Types

- Environment-backed config classes use `pydantic_settings.BaseSettings` with explicit `validation_alias=AliasChoices("ENV_NAME")`; `.env` is loaded at import time.
- Pure shared result types, enums, and constants live under `src/discordbot/typings/` when they do not depend on cogs or utils.
- Use `pydantic.BaseModel` for structured data and type-managed payloads, including service/helper classes. Do not introduce `dataclass`, and do not use loose `object` / `Any` annotations. Keep `Field(description=..., examples=...)` populated on every model. A required field (no default) is written with an explicit ellipsis: `Field(..., description=...)`.

## Coding Conventions

- Ruff is formatter and linter; mypy runs in pre-commit. Use narrow `# noqa: <rule>` comments with a reason.
- Keyword arguments are required for normal calls, including single-argument calls; do not add bare `*` to force keyword-only signatures. Allowed positional idioms: `len(x)`, `str(x)`, `Path("x")`, exception constructors, variadic collectors, `logfire.info("message")`.
- Avoid intermediate one-level aliases such as `usage = responses.usage`.
- Import functions by name and call them unqualified (`from pkg.mod import do_thing`); a module-namespace import is a last resort with a reason.
- LLM latency matters; do not add extra LLM calls for cosmetic improvements.
- `docs/` is generated by `make gen-docs`; do not hand-edit it. Do not touch the README badge block unless explicitly asked.

## Documentation Split

- `README.md` is the concise canonical user-facing README; `README.zh-CN.md` / `README.zh-TW.md` mirror its structure; `CONTRIBUTING.md` is developer-facing English; `CLAUDE.md` stays dense and AI-agent-facing. `AGENTS.md` and `GEMINI.md` are symlinks to `CLAUDE.md`, so edit `CLAUDE.md` only.
